## Safari canonical-host fix

`localhost` and `127.0.0.1` are different cookie origins — a session cookie set on one is invisible on the other. The app's email-confirmation redirect was hardcoded to `APP_URL`'s host (`localhost`), so a Safari session that started on `127.0.0.1` could lose its session after confirming. Fixed with a canonicalizing 301 redirect in `server.js`, applied before authentication middleware, preserving path and query string. Does not weaken cookie security (`httpOnly`/`SameSite`/`Secure` attributes unchanged) — it only ensures a single consistent host is used throughout the journey.

## Stripe webhook and entitlement result

Webhook delivery and processing confirmed working end to end, including recovery when the local Stripe CLI relay dropped a delivery (replayed the real historical event directly against the route with a valid signature — no new Stripe test data created).

A real bug was found and fixed during this testing: `process_stripe_webhook_event` expired **any** active entitlement for a household when **any** of its subscriptions transitioned to a non-qualifying status, without checking that the terminating subscription was the one the entitlement actually referenced. This was caught because the same household ended up with two subscriptions (see open issues below); cancelling the unrelated duplicate would have wrongly expired the household's genuine, paid, active entitlement. Fixed in migration 015 to require `entitlements.external_reference` to match the subscription the event is about before expiring it. Verified fixed: cancelling the duplicate left the real entitlement active and correctly referencing the real subscription.

## Known open issues

- **Duplicate Checkout Session prevention needs investigation.** During testing, a single household ended up with two completed Checkout Sessions/subscriptions. The idempotency-key window in `routes/billing.js` (`POST /billing/create-checkout-session`, 5-minute bucket) should be reviewed to understand how this happened and whether it's sufficient before launch.
- **Migrations 007–009 need database verification.** `007_grant_authenticated_household_reads.sql`, `008_household_isolation_contacts.sql`, and `009_service_role_minimum_app_privileges.sql` are all still headed "DRAFT — NOT APPLIED" in the repo. Given migrations 013/014's headers were found stale in this same session (already applied, headers just not updated), their true state against the live database should be explicitly confirmed rather than assumed in either direction — these cover contacts/household RLS isolation and minimum service-role grants, real security surface for launch.
